# CreativeOS 画布资产工作台产品规划

## 1. 产品定位

CreativeOS 的核心定位是画布资产工作台。
工作台是形式，资产交付是目的

```text
用户用自然语言、参考图或已有节点提出创作意图。
系统判断意图，把结果生成到画布中。
画布中的节点都是可保存、可修改、可追溯、可导出的资产。
```

产品重点不是聊天，也不是单次生成，而是把创作过程沉淀成可交付资产。

## 2. 核心原则

- 画布是主要展现形式，用户默认在画布中完成创作。
- 新建普通画布时不要求用户选择小说、图片、视频等类型。
- 用户先输入意图，系统再判断这是写文章、生图、做视频、处理素材还是自由创作。
- 生成结果必须以节点形式存在于画布中，不能只停留在聊天消息里。
- 节点代表资产，资产可以继续修改、生成下游、导出和复用。
- AI 修改默认生成新版本或新节点，不覆盖原资产。
- 输入区根据当前是否选中节点决定作用范围。

## 3. 主流程

### 3.1 无选中节点

无选中节点时，输入区作用于整个画布。

```text
用户输入：写一篇关于 AI 影视制作的文章
系统判断：文章创作
画布生成：创作意图 -> 文章大纲 -> 文章正文
```

```text
用户输入：生成一张赛博朋克城市图
系统判断：图片生成
画布生成：视觉简报 -> 生图提示词 -> 图片
```

```text
用户输入：做一个 30 秒产品宣传视频
系统判断：视频创作
画布生成：视频简报 -> 脚本 -> 分镜
```

### 3.2 有选中节点

有选中节点时，输入区作用于当前节点。

输入区必须明确显示当前作用对象，例如：

```text
基于「图片 01」继续生成
```

用户可以取消选中，使输入区回到全局模式。

```text
选中图片节点
用户输入：背景更暗一点，角色更突出
系统判断：修改当前图片
画布生成：图片 01 -> 图片 02
```

```text
选中文章正文节点
用户输入：语气更专业一点
系统判断：改写当前文本
画布生成：文章正文 -> 文章正文 v2
```

```text
选中提示词节点
用户输入：改成更适合 Midjourney 的英文 prompt
系统判断：优化提示词
画布生成：生图提示词 -> 英文 Prompt v2
```

## 4. 统一输入区

浮动助手不作为主入口。

第一阶段建议用页面固定输入区承载 AI 操作，可以放在右侧或左侧。具体视觉形式后续再定，但交互职责先明确：

```text
没有选中节点：创建新资产或组织画布内容。
选中节点：基于当前节点调整、变体、下游生成或导出。
```

输入区能力：

- 文本输入。
- 上传参考图。
- 拖入图片。
- 粘贴截图。
- 显示当前选中节点。
- 显示意图识别结果。
- 显示执行状态。

输入区不是普通聊天框，而是创作任务入口。

## 5. 意图判断

每次输入后，系统先判断意图，再决定动作。

动作类型限制在几类：

```text
1. 创建新资产节点
2. 修改当前节点并生成新版本
3. 基于当前节点生成下游节点
4. 导出当前节点或画布资产
5. 回答或解释，不修改画布
```

意图判断结果可以用用户可理解的方式短暂展示：

```text
识别为：图片生成
正在生成视觉简报...
```

```text
识别为：修改图片
正在生成新版本...
```

只有判断不确定时才让用户选择，例如：

```text
你想把这次输入用于：
写文章 / 生成图片 / 做视频
```

## 6. 图片生成示例

### 6.1 只有文字

```text
用户输入：一个赛博朋克女刺客，雨夜霓虹街道，电影感
```

画布生成：

```text
创作意图 -> 视觉简报 -> 生图提示词
```

提示词节点提供：

- 生成图片
- 优化提示词
- 生成变体
- 复制 Prompt
- 导出 TXT

### 6.2 参考图加文字

```text
用户上传参考图，并输入：参考这张图，生成同一角色的战斗姿态
```

画布生成：

```text
参考图 -> 参考分析 -> 生图提示词 -> 图片
创作意图 --------^
```

参考分析要说明系统从参考图中理解到的要素，例如人物外观、服装、配色、构图、风格、保留元素和可变化元素。

### 6.3 已有图片继续修改

```text
用户选中图片节点，输入：保留构图，改成夜晚雪景
```

画布生成：

```text
原图 -> 夜晚雪景版本
```

默认不覆盖原图。

## 7. 修改和版本

不满意时的标准流程：

```text
选中不满意的节点 -> 输入修改要求 -> 生成新版本节点 -> 满意后导出或继续生成
```

默认规则：

- 用户手动编辑文本，可以修改当前草稿。
- AI 生成或 AI 修改，默认生成新版本或新节点。
- 图片、视频、音频等素材默认不覆盖原资产。
- 节点应显示来源关系和版本关系。

## 8. 导出

所有生成结果都必须能成为资产，并具备导出路径。

不同节点的导出方向：

- 文本节点：复制、导出 Markdown、导出 TXT，后续支持 DOCX。
- 提示词节点：复制 Prompt、导出 TXT、导出 JSON。
- 图片节点：导出 PNG/JPG。
- 视频节点：导出 MP4/WebM。
- Sprite 节点：导出帧序列、GIF、WebM、MOV。
- 画布项目：导出资产包、导出画布 JSON。

导出能力要放在节点动作和画布级导出中，不应隐藏在聊天消息里。

## 9. 自由画布入口

需要保留一个自由画布入口，面向专业用户。

自由画布定位：

```text
专业用户可以自由放节点、手动连接节点、选择节点类型、直接调用 AI 辅助。
```

自由画布适合：

- 用户清楚自己要创建什么节点。
- 用户知道节点之间应该如何连接。
- 用户希望手动搭建生成链路。
- 用户需要更接近 ComfyUI / Figma / 白板式操作的专业模式。

产品入口规划：

- 首页主入口：新建画布，进入意图驱动的资产生成画布。
- 首页次级入口：自由画布，面向专业用户，后续实现。

自由画布不是第一阶段主流程，但要在产品规划中保留。

## 10. 第一阶段建议

第一阶段只做最小闭环：

```text
新建画布
-> 进入画布
-> 固定输入区输入文字或上传参考图
-> AI 判断意图
-> 生成节点
-> 选中节点后继续修改
-> 节点可导出
```

优先验证图片生成链路：

```text
文字 / 参考图
-> 视觉简报
-> 生图提示词
-> 图片
-> 图片变体
-> 导出 PNG
```

后续再扩展文章、小说、视频和 Sprite。

当前文本资产链第一版已经覆盖：

```text
文章：创作意图 -> 文章大纲 -> 文章正文
角色：创作意图 -> 角色简报 -> 角色卡
剧本：创作意图 -> 剧本结构 -> 剧本正文
分镜：创作意图 -> 分镜方案 -> 分镜表
小说：小说意图 -> 作品定位 / 世界观简表 / 角色种子 / 主线大纲
章节：章节意图 -> 章节大纲 -> 章节正文
```

这些链路仍然来自用户意图判断，不需要用户在新建画布时选择类型。

## 11. 记忆系统

记忆系统服务于意图判断、上下文选择、资产一致性和长期项目管理。

记忆系统不等于固定工作流，也不应该让用户按步骤推进。用户仍然只通过画布输入意图，AI 根据当前画布、选中节点、历史输入和项目记忆判断下一步。

### 记忆分层

1. 画布事实

画布节点、连线、资产、版本关系是最权威的事实源。

2. 当前线程记忆

保存当前这次输入上下文、最近追问、待确认动作和选中节点。它只影响短期连续交互，例如用户说“继续”时如何判断目标。

3. 项目记忆

对一个画布项目长期有效。小说、长文章、视频系列尤其依赖这层。

典型项目记忆：

- 作品定位
- 世界观规则
- 角色状态
- 已发生事件
- 伏笔和待回收信息
- 用户确认过的主版本
- 项目资产索引

当前自动同步的项目记忆：

- 项目资产索引：从画布资产和重要文本节点生成。
- 上次沟通摘要：从最近沟通、近期资产和上次焦点生成。
- 作品设定类记忆：从标题包含作品、定位、世界观、规则、设定、大纲、主线、风格的文本节点生成。
- 角色状态类记忆：从角色、人物、关系、人设类文本节点生成。
- 连续性类记忆：从连续性、伏笔、章节事件、已发生事件类文本节点生成。

记忆召回不能只按更新时间。长期可用的召回策略应综合：

- 当前用户输入的文本相关性。
- 当前选中节点或上传参考节点。
- 记忆类型权重。
- 记忆重要性。
- 记忆置信度。
- 更新时间和最后访问时间。

当前第一版已经使用文本相关性、焦点节点、类型权重、重要性、置信度、新近度和 embedding 相似度排序；后续再接 pgvector / 向量索引做生产级语义检索。

4. 用户记忆

跨项目复用，例如用户偏好的风格、语言、输出长度、常用创作方向。

5. 系统规则

系统规则不应由 AI 随意改写，例如：

- 低置信度不生成节点。
- 普通问题默认不写入画布。
- AI 修改默认生成新版本。
- 前端只执行白名单动作。

### 意图门控

用户输入后先做门控判断：

```text
create_asset
modify_asset
ask_question
navigate_canvas
unclear
```

如果置信度低于阈值，不创建节点，只追问。

普通问题或外部问题默认只回答，不写入画布。只有用户明确要求“生成、写进、放到画布、基于这个创建资产”时，才进入画布执行。

项目记忆会进入两个位置：

- 意图判断：帮助 AI 判断用户是在问问题、修改资产还是创建资产。
- 文本生成：帮助正文、设定、角色、提示词等文本资产保持上下文一致。

项目记忆只作为上下文，不代表用户当前明确指令。当前输入和选中节点优先级更高。

### 小说场景

小说不是线性流程，而是长期资产图。

小说相关记忆重点：

- 作品圣经
- 连续性记录
- 角色状态表
- 章节事件摘要
- 伏笔台账

写正文时，AI 不应该读取全画布，而是选择相关记忆和节点：

```text
作品定位
相关角色
当前章节目标
连续性记录
用户当前要求
```

写完章节后，后台提取并更新连续性记录、角色状态和伏笔变化。

当前第一版已在章节正文生成后提取并写入连续性记录；角色状态和伏笔台账仍需后续细分。

### 技术原则

- 项目记忆必须存数据库。
- 记忆写入必须结构化，不能只靠聊天记录。
- AI 只能返回记忆补丁或计划，服务端校验后再写库。当前已有项目记忆 patch 写入 API，后续需要接后台提取任务。
- 热路径只做必要读写，复杂记忆提取可以后台完成。
- LangGraph 用于编排和短期线程状态，Postgres 作为长期记忆与资产事实源。
- 大文件不应该长期以 data URL 存在项目 JSON 里。当前已新增本地上传底座，画布新上传媒体和新生成图片/视频都会保存为文件 URL，并写入文件归属记录；上传文件和生成 data URL 落盘统一走 `canvas-asset-storage` 存储层；删除项目时会清理已关联的本地文件；未引用文件可通过后台清理接口定期删除。后续云部署必须扩展对象存储驱动，画布只保存文件 URL、元数据和资产引用。

### 再次进入画布

用户下次进入同一个画布时，需要续上原来的沟通上下文。

恢复依据不是无限聊天记录，而是三类数据：

- 画布事实：节点、连线、资产、版本关系。
- 会话摘要：上次沟通重点、近期资产、上次焦点。
- 项目记忆：资产索引、长期设定、连续性记录。

进入画布后，系统应恢复上次焦点节点。用户继续输入时，AI 先读取会话摘要和项目记忆，再做意图判断。

如果用户问“继续”“刚才那个再改一下”，不能只靠字面触发，而要结合上次焦点、最近沟通摘要和当前选中节点判断。判断不明确时追问，不直接生成节点。
